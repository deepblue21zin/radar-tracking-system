## Update 1

### 1. `requirements.txt` 재정리
- 기존 상태:
  - conda 환경 전체를 `pip freeze`한 형태라서 unrelated 패키지가 많이 섞여 있었다.
  - `file:///...` 형태의 로컬 빌드 경로와 `cp39` wheel 경로가 포함되어 있어서 다른 PC나 Python 3.11 환경에서 그대로 재현하기 어려웠다.
- 수정 내용:
  - 프로젝트 런타임에 실제로 필요한 최소 패키지만 남겼다.
  - 현재 내용은 아래 4개다.
    - `pyserial==3.5`
    - `numpy==1.26.4`
    - `scikit-learn==1.5.2`
    - `filterpy==1.4.5`
- 수정 이유:
  - 팀원이 같은 환경을 다시 만드는 데 필요한 파일로 바꾸기 위해서다.
  - "내 PC에서는 되는데 팀원 PC에서는 안 되는" 문제를 줄이기 위한 정리다.

### 2. `src/parser/tlv_parse_runner.py` cfg 응답 로그 보강
- 기존 상태:
  - cfg 파일 내용을 CLI 포트로 전송만 하고, 보드가 `Done`인지 `Error`인지 읽지 않았다.
  - 그래서 cfg/demo mismatch가 있어도 현장에서 원인 파악이 느릴 수 있었다.
- 수정 내용:
  - cfg 명령 전송 시 `[CFG] >> ...` 로그를 출력하도록 변경했다.
  - CLI 포트 응답을 읽어서 `[CFG] << ...` 로그를 출력하도록 변경했다.
  - 응답에 `Error` 또는 `Fail`이 포함되면 즉시 예외를 발생시키도록 보강했다.
- 수정 이유:
  - 보드 bring-up에서 "왜 데이터가 안 올라오는지"를 빠르게 확인하기 위해서다.

### 3. `src/parser/tlv_packet_parser.py` `Type 1` 누락 frame fail 처리
- 기존 상태:
  - `num_det_obj > 0`이어도 `Type 1` payload가 없는 프레임을 명시적으로 실패시키지 않았다.
  - 이런 경우 뒤 단계에서 0값으로 채워진 point가 정상 데이터처럼 보일 위험이 있었다.
- 수정 내용:
  - TLV loop 중 `Type 1`을 실제로 봤는지 추적하는 플래그를 추가했다.
  - `num_det_obj > 0`인데 `Type 1`을 한 번도 못 보면 해당 프레임을 `FAIL` 처리하도록 수정했다.
- 수정 이유:
  - parser는 프레임 단위로 데이터 유효성을 판정해야 한다.
  - 연속 횟수 기반보다 "그 프레임이 유효한가"를 바로 판단하는 쪽이 더 안전하다.
