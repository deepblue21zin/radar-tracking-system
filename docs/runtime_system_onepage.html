1. End-to-End Flow

시스템은 TLV packet을 읽고, point를 world 좌표로 보정한 뒤, 필터링과 DBSCAN, Kalman tracking, control 판단, logging/reporting, viewer 렌더링으로 이어진다. viewer는 계산을 따로 하지 않고 runtime 결과를 그대로 본다.

Step 1

UART Read

레이더 CLI/Data 포트에서 바이트 스트림을 읽는다.

src/parser/runtime_pipeline.py / MMWaveSerialReader
Out: raw byte buffer
Step 2

TLV Parse

magic word를 찾고 TLV packet을 해석해 frame 단위 구조로 만든다.

src/parser/tlv_packet_parser.py
Out: ParsedFrame(frame_number, num_obj, points, packet_bytes, ...)
Step 3

World Transform

sensor yaw/pitch/height를 이용해 sensor frame point를 world frame으로 바꾼다.

src/parser/runtime_pipeline.py / transform_points_to_world()
Out: world-frame point list with x, y, z, ground_range
Step 4

Preprocess

SNR, noise, range, ROI, keepout, static clutter filter를 적용한다.

src/filter/noise_filter.py / preprocess_points()
Out: filtered points + FilterStats
Step 5

DBSCAN

filtered point를 cluster로 묶고 centroid/confidence를 계산한다.

src/cluster/dbscan_cluster.py / cluster_points()
Out: cluster list {x,y,z,v,size,confidence,...}
Step 6

Tracking

cluster를 track로 이어서 위치/속도 상태를 유지한다.

src/tracking/kalman_tracker.py / MultiObjectKalmanTracker
Out: TrackOutput(track_id,x,y,vx,vy,...)
Step 7

Control / Reports / Viewer

belt control 판단, CSV/log/report 생성, live viewer 렌더링으로 나뉜다.

src/control, src/reporting, src/visualization
Out: decision, run artifacts, live UI

2. Visual Reading Guide

이 시스템은 크게 두 덩어리로 보면 이해가 쉽다. 센서 내부 firmware/DSP 쪽에서 point cloud를 만들고, Python host 쪽에서는 그 point cloud를 해석하고 정리하고 추적하고 보여준다.

Sensor Side: mmWave Demo Firmware

1. Chirp / Frame 설정

2. CFAR / FOV

3. Beamforming / AoA
multiObjBeamForming, aoaFovCfg로 angle 해석과 다중 반사 분리를 수행한다.
4. TLV Output
guiMonitor로 point cloud와 side info를 UART TLV로 내보낸다.

Host Side: Python Runtime

1. Parse

3. Preprocess / DBSCAN / Tracking

같은 runtime 결과를 viewer, belt control, CSV/markdown report에 공유한다.
꼭 기억할 한 줄
이 프로젝트는 raw ADC에서 시작하지 않는다. 센서가 만든 detected point cloud부터 Python 파이프라인이 시작된다. 그래서 beamforming/CFAR 자체를 Python에서 다시 하는 것이 아니라, 그 결과를 설치 보정/필터/클러스터/트래킹으로 다듬는 구조다.

증상에서 코드 찾기

run_viewer(), _runtime_worker(), draw_snapshot(), 그리고 disable_text_log/max_vis_fps를 함께 본다.

3. Code Roles

아래 카드만 보면 어떤 파일이 시스템에서 무슨 책임을 갖는지 빠르게 파악할 수 있다.

Entrypoint

src/parser/tlv_parse_runner.py

사용자가 직접 실행하는 기본 명령 진입점이다. 실제 구현은 runtime pipeline으로 넘기고, 기존 import 호환도 유지한다.

Runtime Core

src/parser/runtime_pipeline.py

실제 메인 루프. serial read, frame processing, logging, control, experiment report, performance log까지 모두 묶는다.

Parser

src/parser/tlv_packet_parser.py

TLV packet을 해석해 dict-of-arrays 형태의 point 데이터를 뽑아낸다.

Filter

src/filter/noise_filter.py

point quality와 geometry 기반 전처리를 수행하고, 어떤 기준으로 얼마나 제거됐는지 FilterStats로 남긴다.

Cluster

src/cluster/dbscan_cluster.py

point를 object 후보 단위로 묶는다. 현재 사람 분할 이슈와 가장 직접적으로 연결되는 모듈이다.

Tracker

src/tracking/kalman_tracker.py

cluster centroid를 track로 이어서 상태를 예측/갱신한다. 현재는 greedy nearest-first association을 사용한다.

Control

src/control/proximity_speed_control.py

track 또는 cluster를 zone과 belt axis 기준으로 해석해서 RESUME / SLOW / STOP을 결정한다.

MCU TX

src/communication/control_protocol.py

Python host에서 STM32로 보낼 제어 packet을 인코딩하고 serial heartbeat 포함 송신을 관리한다.

Viewer

src/visualization/live_rail_viewer.py

runtime worker가 만든 최신 snapshot만 받아 시각화한다. 계산을 따로 하지 않고 draw만 담당한다.

Params

src/runtime_params.py / config/runtime_params.json

공용 기본값과 JSON override를 담당한다. runner와 viewer가 같은 기본값을 공유한다.

Experiment Report

src/reporting/runtime_experiment_report.py

run 종료 시 문제/가설/수정/검증 템플릿이 포함된 run별 markdown report를 생성한다.

Performance Log

src/reporting/performance_log_report.py

run_summary.csv와 frame CSV를 읽어 최신 상태와 이전 run 대비 변화가 보이는 자동 성능 로그를 생성한다.

            if d <= self.association_gate:
                candidates.append((d, ti, mi))
    candidates.sort(key=lambda x: x[0])

Control control_belt_axis_x/y, approach_speed_threshold, stationary_speed_threshold, static_hold_sec belt 방향 기준 속도/정지 판정을 안정화한다. STOP/SLOW/RESUME가 채터링할 때
Viewer max_vis_fps, point_persistence_frames, track_history_sec, velocity_arrow_scale 시각적 잔상, trail, 체감 부하를 바꾼다. 화면이 늦거나 너무 복잡해 보일 때
Health Metrics parse_failures, resync_events, dropped_frames_estimate UART/parser 안정성을 본다. point가 중간중간 사라질 때

7. Current Problems

지금 남아 있는 핵심 문제는 한 가지가 아니라, 좌표계·필터·클러스터링·연속성 문제가 겹친 상태다.

Issue 1

대각선처럼 보이는 이동

  • 원인 1: 설치 yaw/pitch/height와 world 좌표계가 정확히 맞지 않으면 앞뒤 이동이 x/y/z에 섞인다.
  • 원인 2: 사람이 여러 track으로 분할되면 viewer에서는 대각선/지그재그처럼 더 과장돼 보인다.
  • 원인 3: range/ROI로 몸 일부만 살아남으면 centroid가 좌우로 흔들린다.
Issue 2

사람 1명이 여러 track으로 보임

  • mmWave point cloud가 원래 성기고, 몸 좌/우/팔 반사가 따로 뜬다.
  • DBSCAN 파라미터가 보수적이면 같은 사람 안의 빈 공간을 못 메운다.
  • tracker가 greedy nearest-first라 cluster가 둘 이상 뜨면 track도 둘 이상 유지되기 쉽다.
Issue 3

Zero-Track 긴 구간

  • range gate나 ROI가 멀어진 구간에서 사람 점군을 자른다.
  • cluster가 한번 안 생기면 tracker miss가 연속으로 누적된다.
  • 현재 화면엔 최신 상태만 보여주므로 실제 미검출 streak가 그대로 드러난다.
Issue 4

Parser / UART 연속성

  • magic-word resync와 dropped frame은 줄었지만 완전히 사라지진 않았다.
  • viewer draw stall보다 parser가 더 나쁘면 UART continuity가 근본 병목이다.
  • runner-only run과 viewer run을 분리해서 봐야 원인을 정확히 나눌 수 있다.

8. What To Solve Next

문제별로 손대야 할 위치가 다르다. 아래 순서대로 가는 것이 지금 구조에서는 가장 효율적이다.

Priority 1

Range Gate를 Ground-Centric하게 바꾸기

  • 현재 point는 world frame으로 보정되지만 filter의 range 판단은 여전히 기존 range 의존이 크다.
  • ground_range = hypot(x, y)를 filter의 선택적 기준으로 도입하면 상부 설치에서 더 직관적인 gate가 된다.
  • 앞뒤 데모라면 max_range보다 y ROI + ground_range 조합이 더 맞다.
Priority 2

사람 분할 줄이기

  • near-range band의 eps를 키우거나 use_velocity_feature를 실험한다.
  • 서로 가까운 cluster를 post-merge하는 후처리를 넣는다.
  • viewer에는 primary track 우선 표시를 넣어 시각적 분할도 줄일 수 있다.
Priority 3

Zero-Track 완화

  • report_miss_tolerance를 올려 화면 깜빡임을 줄인다.
  • max_misses를 조금 늘려 short gap을 버티게 한다.
  • 다만 miss를 너무 키우면 ghost track가 늘 수 있으므로 experiment report로 같이 봐야 한다.
Priority 4

UART Continuity 더 줄이기

  • 현재 viewer는 read/process와 draw를 분리했으므로 다음 기준점은 runner-only run이다.
  • runner에서도 resync가 계속 크면 reader 정책, packet batch 처리, serial timeout 조정이 다음 대상이다.
  • 이건 frame CSV보다 parse_fail / resync / dropped 변화로 보는 것이 맞다.
현재 가장 합리적인 다음 단계
ground_range 기반 range gate 도입 → near-range cluster merge 또는 velocity feature 실험 → tracker visibility 완화 → runner-only 기준 UART continuity 재측정

9. Does Auto Recording Slow The System?

자동 기록은 종류마다 부담이 다르다. 전부 같은 비용이 아니고, 특히 debug와는 성격이 다르다.

Low / End Of Run

Experiment Report / Performance Log

이 둘은 run이 끝난 뒤 run_summary.csv와 frame CSV를 읽어서 생성된다. 실시간 frame loop에서 매번 돌지 않는다.

결론: 시연 중 딜레이 원인으로 보기는 어렵다.

Medium

CSV / Text Log Flush

frame CSV와 특히 text log는 per-frame 또는 per-line flush가 있어 부담이 있다. 이건 실제 runtime loop 비용에 영향을 줄 수 있다.

대응: --disable-text-log, 필요시 --disable-file-log.

High

Debug Output

--debug는 parser 내부 상태를 콘솔에 많이 뿌려서 훨씬 무겁다. 자동 report와는 비교 대상이 아니다.

결론: debug는 진단용, 시연용이 아니다.

실전 해석
자동 experiment report와 performance log는 종료 시 생성이라 실시간 영향이 작다. 시연 중 체감 지연을 만드는 건 대체로 text log flush, viewer draw, UART continuity 쪽이다. 그래서 viewer는 기본적으로 file log를 끄고, runner-only 비교 run에서는 --disable-text-log --disable-overview-png를 먼저 쓰는 것이 맞다.
